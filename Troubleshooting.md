# **Troubleshooting (Решение типичных проблем)**

В этом файле собраны архитектурные решения инфраструктурных проблем, возникающих при локальном развертывании в связке **Windows + Docker Desktop**.

## **1. Зависание Composer (I/O Hang в WSL 2)**

* **Симптомы:** Консоль намертво зависает при composer install внутри контейнера.  
* **Причина:** Дедлок файловой системы моста 9p, связывающего NTFS и Linux при активном I/O.  
* **Решение в проекте:** Зависимости Composer скачиваются во временную директорию /build на этапе сборки Dockerfile, а не монтируются напрямую из ОС.

## **2. Ошибка 504 Gateway Time-out при установке Битрикс**

* **Симптомы:** На шаге "Установка главного модуля" процесс прерывается 504 ошибкой Nginx.  
* **Причина:** Распаковка архива ядра Битрикса (сотни тысяч файлов) работает крайне медленно из-за низкой скорости записи при монтировании Volume в Windows.  
* **Решение в проекте:** В конфигурации Nginx (fastcgi_read_timeout) и PHP (max_execution_time) установлены лимиты в 600 секунд (10 минут).

## **3. "Время отличается на 10800 секунд" (Проверка системы)**

* **Симптомы:** Красная ошибка в административной панели Битрикса (Проверка системы -> Сокеты / Время).  
* **Причина:** Официальные образы MariaDB работают в часовом поясе UTC, а PHP в Europe/Moscow (+3 часа).  
* **Решение в проекте:** В docker/mysql/bitrix.cnf жестко задан параметр default-time-zone = '+03:00'.

## **4. Падение Master/Slave Базы Данных (GTID)**

* **Симптомы:** Контейнеры MariaDB падают с ошибкой unknown variable 'enforce-gtid-consistency=ON'.  
* **Причина:** Использование флагов настройки GTID от MySQL 8.x в MariaDB 10.x.  
* **Решение в проекте:** Переход на нативный синтаксис MASTER_USE_GTID=slave_pos в init.sql.

## **5. Воркер RabbitMQ падает при старте (Connection Refused)**

* **Симптомы:** microservice_worker уходит в циклический рестарт.  
* **Причина:** Воркер стартует быстрее, чем RabbitMQ успевает открыть TCP порт.  
* **Решение в проекте:** Настроен healthcheck для RabbitMQ, а воркер ожидает condition: service_healthy.

## **6. Битрикс не видит микросервис (Rejected code U+0000005F)**

* **Симптомы:** Ошибка D7 Error converting hostname to punycode: Rejected code U+0000005F при HTTP-запросе к API.  
* **Причина:** Символ подчеркивания (_) в имени Docker-контейнера (microservice_api). Стандарты интернета (RFC 952) строго запрещают использовать _ в доменных именах, и HttpClient Битрикса блокирует такие запросы на этапе валидации.  
* **Решение в проекте:** Контейнер переименован в microservice-api (через дефис).

## **7. Фоновый воркер не видит изменения в коде**

* **Симптомы:** Вы отредактировали PHP-файл воркера (например, изменили путь сохранения отчета), но воркер продолжает работать по старой логике.  
* **Причина:** Демонизированные CLI-процессы (worker.php с бесконечным циклом while) загружают классы в оперативную память при старте. Они не перечитывают файлы с диска при каждом сообщении.  
* **Решение в проекте:** Любое изменение кода бизнес-логики, которую использует воркер, требует его перезапуска: docker-compose restart microservice_worker.

## **8. Ошибка сохранения файла (No such file or directory)**

* **Симптомы:** Воркер выбрасывает Warning: fopen(/var/www/html/upload/...): Failed to open stream.  
* **Причина:** Изолированный контейнер микросервиса не имел доступа к директории /upload/ монолита.  
* **Решение в проекте:** Папка src/bitrix/upload примонтирована (bind mount) в контейнеры микросервисов в docker-compose.yml. В сервисе генерации отчетов добавлена проверка is_dir() и создание папки через mkdir($dir, 0775, true).
